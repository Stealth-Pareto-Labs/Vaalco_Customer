# Navigator Z — Vessel Intelligence

An offshore vessel monitoring platform with **two halves that share one brain**:

- **Ask** — a chatbot. You ask a question ("Why was fuel high on the 22nd?")
  and it answers, powered by a **frontier model via the OpenAI API (GPT-4o)**.
- **Signals** — an automated intelligence layer. Every time a new daily report
  arrives, it scans the data on its own, decides what matters (low / medium /
  high), explains each concern with the evidence behind it, recommends next
  steps, and **emails and texts the report to the right people** — no one has
  to open the app.

Both read the **real multi-sheet midnight reports as-is** (DPR, Technical Info,
HSE Statistics) — preserving both the structured numbers and the unstructured
hour-by-hour activity narrative — and both run on the **same analysis engine**,
so a number in the chatbot and a number in the intelligence report can never
disagree.

The design principle throughout: **the code does all the math, the model does
the language.** Rules decide *whether* something is a concern and *how severe*
it is; the model only phrases it for a busy superintendent. Every figure comes
from the deterministic analysis code — never invented by the model.

## ⚠️ The one thing you need to add: your API key

Open **`app/config.py`** and find this line near the top (section 1):

```python
OPENAI_API_KEY = os.environ.get("OPENAI_API_KEY", "")   # <-- paste your key inside the quotes
```

Paste your OpenAI API key between the empty quotes, so it reads:

```python
OPENAI_API_KEY = os.environ.get("OPENAI_API_KEY", "sk-YOUR-KEY-HERE")
```

Get a key at <https://platform.openai.com/> → API keys → Create new key.

*(Preferred for anything beyond a local demo: leave that line alone and instead
run `export OPENAI_API_KEY="sk-..."` in your terminal before starting. The
environment variable overrides the file.)*

That is the only change needed to make **both** systems fully operational. The
platform will even run **without** a key — the intelligence engine falls back to
plain-English summaries generated by the rules alone, so nothing breaks — but
the AI narration and the chatbot need the key.

## Run it

```bash
pip install -r requirements.txt      # one dependency: openpyxl
python app/server.py                 # from the project root
```

Then open **<http://localhost:8000>**. You'll land on **Ask**; click **Signals**
for the intelligence console.

The only runtime dependency is `openpyxl` (to read the Excel reports).
Everything else — the web server, the email sender, the SMS sender — uses the
Python standard library. No build step, no framework, no database.

## How the two halves connect

- One shared engine (`analysis.py`) computes every number. **Ask** and
  **Signals** are just two ways of consuming it.
- Every signal card in the console has an **"Ask about this"** button. It drops
  a ready-made question into the Ask tab and runs it — so a flagged concern is
  one click from a full conversational deep-dive.
- When a new `.xlsx` lands in `reports/` while the app is running, **both**
  react: the chatbot ingests it (and says so), and the intelligence engine runs
  a fresh analysis and sends the report.

## What the intelligence engine looks for

Detection is rule-based and lives in `app/signals.py`. Out of the box it watches:

| Area | What it flags | Severity logic |
|------|---------------|----------------|
| **Fuel** | A day burning more than the workload model predicts; a sharp step-up vs recent days; the worst outstanding day in the window | By how many standard deviations it exceeds the model, and the % jump |
| **DP efficiency** | A wide spread between the most and least efficient DP days | By the spread % |
| **Maintenance** | Machines overdue for a lube-oil service, or due soon | Overdue = always high; else by days remaining |
| **Engine** | Telemetry that is hand-entered rather than measured (can't be trended) | Low (a data-quality note) |
| **Fluids / tanks** | Consumables running toward empty; waste tanks genuinely filling up | By days-to-empty / rising trend |
| **HSE** | A near miss or incident logged today | Configurable |

Every threshold is tunable in **`app/config.py`** section 5
(`SIGNAL_THRESHOLDS`) — operations can recalibrate what counts as high/medium
without touching detection code.

> The engine is deliberately careful about false alarms. For example, a waste
> tank sitting at a flat level (common when logs are hand-entered) is **not**
> flagged as "filling up" — only a genuine upward trend is. Likewise, engine
> readings that are identical across every report are surfaced as a
> *data-quality* note, not a fake health trend.

## Delivery: email and SMS

Configured in **`app/config.py`**. Both are **safe by default** — on a fresh
checkout with nothing filled in, the system writes exactly what it *would* have
sent to `reports_out/outbox/` and logs it. Nothing is sent, nothing crashes, and
no paid message goes out until you opt in. This lets you verify the whole
pipeline before wiring up real credentials.

### Email (SMTP) — `config.py` section 2

Works with any provider (Gmail, Office 365, SendGrid SMTP, Amazon SES SMTP, …):

```python
SMTP_HOST = "smtp.gmail.com"     # your mail server
SMTP_PORT = 587                  # 587 = STARTTLS (typical)
SMTP_USER = "you@company.com"    # login
SMTP_PASSWORD = "app-password"   # login password / app password
EMAIL_FROM = "vessel-intelligence@company.com"
```

Leave `SMTP_HOST` blank to keep email in "save to outbox" mode.

### SMS (Twilio) — `config.py` section 3

**Off by default.** To enable, create a Twilio account and fill in:

```python
SMS_ENABLED = True
TWILIO_ACCOUNT_SID  = "AC..."
TWILIO_AUTH_TOKEN   = "..."
TWILIO_FROM_NUMBER  = "+1..."    # your Twilio number
```

SMS uses Twilio's REST API directly (no SDK). While `SMS_ENABLED` is `False` or
the credentials are blank, the digest is saved to the outbox instead of sent.

### Recipients — `config.py` section 4

Who gets the report, and who gets a text:

```python
RECIPIENTS = [
    {"name": "Marine Superintendent",
     "email": "superintendent@company.com",
     "phone": "+1...",
     "sms_from_priority": "high"},     # texted only for HIGH signals
    {"name": "Fleet Analyst",
     "email": "analyst@company.com",
     "phone": "+1...",
     "sms_from_priority": "medium"},   # texted for MEDIUM and HIGH
]
```

Everyone listed gets the full email. SMS is gated per person by
`sms_from_priority` (`"high"`, `"medium"`, or `"none"`), so management can get a
text only when something is genuinely high-priority while an analyst gets more.

## Using the Signals console

- **Re-run** — run the analysis again against the current data and save it to
  history.
- **Preview email** — open the exact branded report that gets emailed, in a new
  tab (useful for print or a quick look).
- **Send now** — deliver the current report over email + SMS immediately
  (respects the safe-by-default rules above).
- **Report history** — every run is saved to `reports_out/` as JSON; click any
  row to reload that report.

## Project layout

```
app/
  server.py        ← ENTRY POINT. Serves the two-tab app + all endpoints.
  config.py        ← EVERYTHING you configure: API key, SMTP, SMS, recipients,
                     thresholds. Start here.

  # shared core (used by BOTH Ask and Signals)
  parser.py        ← reads the multi-sheet Excel reports
  analysis.py      ← all deterministic maths + the fuel model + data store
  claude_client.py ← the chatbot's tool-calling loop (OpenAI)

  # the intelligence layer
  signals.py       ← rule-based detectors (decide IF + severity)
  intelligence.py  ← runs a full analysis, adds AI narration, saves history
  report.py        ← renders the branded HTML email, plain text, and SMS digest
  notify.py        ← sends email (SMTP) and SMS (Twilio); safe by default
  watcher.py       ← watches reports/ and triggers a run on each new report

reports/           ← drop daily .xlsx reports here (8 sample reports included)
reports_out/       ← generated at runtime: saved runs + outbox/ (created on first run)
assets/            ← VAALCO logo used in the email report
```

## A note on trust

This platform is built for a client who reasonably distrusts "AI that makes up
numbers." That is why the split is strict and visible:

- **The rules** decide what is a concern and how serious it is. They are plain
  Python you can read in `signals.py`, and every threshold is in `config.py`.
- **The model** only writes the executive summary, the plain-English
  explanation, and the suggested next steps — always from evidence the code
  already computed. It is explicitly instructed never to change a number, a
  date, or a severity.
- Every emailed report says, in its footer, whether it was narrated by the AI or
  produced by the rules alone, and states that all figures are computed by the
  engine.

If the model is ever unavailable, the system still runs and still sends — it
just uses the rules' own wording. The intelligence never depends on the model
being reachable.
